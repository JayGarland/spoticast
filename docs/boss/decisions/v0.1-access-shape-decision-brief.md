# v0.1 Access Shape Decision Brief

Audience: Boss

## Summary

The internal product reviewer found that Resonova's top v0.1 blocker is no longer missing playback controls. The current blocker is the gap between:

```text
Current developer-stage access:
clone repo -> configure .env -> provide API keys -> run local server -> connect Spotify
```

and:

```text
Intended v0.1 access:
open Resonova -> connect Spotify Premium -> generate/listen directly
```

This brief frames the access-shape decision. It does not approve implementation, hosting, billing, cloud accounts, subscriptions, or release architecture.

## Current Facts

- Resonova is currently usable as a developer/owner MVP.
- Normal users cannot reasonably use the current setup because it requires local server setup, ffmpeg, `.env`, Spotify configuration, Gemini credentials, and command-line operation.
- The stated v0.1 direction is direct use after Spotify connection, with no user-managed AI API keys.
- AI generation and TTS are not free, so v0.1 access shape eventually intersects with cost recovery.
- The durable product wedge is inspectable personal music memory that can steer hosted AI commentary, not self-hosting.
- Persistent memory is not implemented yet, so v0.1 copy must not overpromise learning unless the minimal profile layer ships first.

## Access Shape Options

| Option | Shape | User fit | Engineering cost | Product risk | Chef read |
|---|---|---:|---:|---:|---|
| A. Developer-only alpha | Keep repo/local setup as the only supported path | Low for normal users | Low | High if called v0.1 | Good for current development, not v0.1 |
| B. Private hosted owner/beta instance | Resonova runs as a hosted/private app; user connects Spotify; platform-owned AI keys | High | Medium-high | Medium | Best candidate for v0.1 direction discussion |
| C. Public hosted app with accounts/billing | Full normal-user product with accounts, payment, quotas, hosted storage | High | Very high | Very high | Too much before core reliability/memory decisions |
| D. Self-hosted product positioning | Market Resonova as bring-your-own-server/API-key | Low-medium | Low-medium | High | Contradicts current boss steering unless explicitly chosen |

## Recommended Decision Direction

Recommended discussion target:

```text
Option B: private hosted owner/beta instance as the v0.1 north star.
```

Why:

- It matches the product direction: connect Spotify and use directly.
- It avoids asking normal users for Gemini/TTS API keys.
- It can remain private or invite-only while reliability, cost, and memory behavior are validated.
- It does not require solving full public billing, multi-user scale, or broad account architecture immediately.
- It keeps self-hosting as a developer-stage path, not the product promise.

This is not implementation approval. It is the access-shape hypothesis chef would use for the next strategy discussion.

## Decisions Needed Before Build

Boss-chef decisions required before any manager implementation:

1. Is v0.1 intended to be **private/invite-only hosted beta** rather than public launch?
2. Who pays AI/TTS costs during v0.1 validation: boss budget, limited free allowance, or another temporary cap?
3. Is Spotify the only required connected source for v0.1?
4. Should v0.1 memory be local-to-instance for the owner/beta or designed with future hosted-user storage in mind?
5. Should the landing copy be softened now, or should minimal memory be prioritized so the claim becomes true?
6. What is the minimum user audience for v0.1: boss only, a small tester group, or broader private beta?

## Immediate Non-Architecture Work

The following can be discussed as scoped cleanup while access architecture remains undecided:

- Fix shared outro path so saved episodes replay their own outro.
- Make landing/README copy honest about memory until profile exists.
- Remove end-user-facing developer setup text from the app surface, or label it clearly as developer-stage.
- Fix Windows/cross-platform setup and paste hints.
- Verify reduced-motion/accessibility polish.

These are small correctness/honesty tasks, not v0.1 architecture.

## Parked Until Explicit Approval

- Public account system.
- Billing, subscription, quotas, free trials, or payment integration.
- Hosted persistent memory storage architecture.
- Multi-source support beyond Spotify.
- Native app or PWA packaging.
- Public launch positioning.
- Treating self-hosting as the main market promise.

## Chef Recommendation

Use this decision brief to discuss whether **private hosted beta** is the right v0.1 access target. In parallel, allow a small correctness/honesty cleanup if boss wants quick trust wins.

Do not assign cloud accounts, billing, multi-source, or hosted-memory implementation until the above decisions are explicit.
