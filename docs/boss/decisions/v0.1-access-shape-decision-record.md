# Decision Record - v0.1 Access Shape

Audience: Internal

## Decision

Use a **private hosted owner/beta instance** as the planning north star for Resonova v0.1.

The intended v0.1 user experience is:

```text
open Resonova -> connect Spotify Premium -> generate/listen directly
```

Normal v0.1 users should not need to:

- clone the repository
- run a local server
- configure `.env`
- provide Gemini/TTS API keys
- understand developer setup

## Meaning

This decision records product direction only. It does **not** approve implementation of hosting, accounts, billing, subscription, public launch, hosted memory storage, or release architecture.

For now:

- current local setup remains valid for boss/developer-stage work
- self-hosting is not the main product promise
- Spotify remains the required v0.1 source unless boss reopens the decision
- AI/TTS cost recovery remains parked
- public accounts and billing remain parked

This decision does not de-scope Resonova's origin goal. The product should still move toward persistent personal memory, user profile, feedback, and Spotify-authorized listening trails so the companion understands the user better with continued use.

## Why

Accepted inputs:

- `docs/handoffs/Product Baseline Release Readiness Review.md`
- `docs/boss/decisions/v0.1-access-shape-decision-brief.md`
- `docs/boss/decisions/release-access-and-memory-positioning-brief.md`
- `docs/boss/decisions/attention-board-next-moves-brief.md`

Reasoning:

- The current product is good enough as a developer/owner MVP.
- The top v0.1 blocker is access/onboarding mismatch, not missing playback controls.
- A normal user cannot reasonably run the current local setup.
- A private hosted beta preserves direct-use product direction without forcing public launch, billing, or large account architecture now.
- The direct-use entry path is compatible with memory: a new test user should be able to open the browser site, connect Spotify, authorize Resonova, and begin building a profile through listening and feedback.
