# Resonova v0.1 Roadmap

Audience: Boss

## Summary

Resonova v0.1 should move from a developer-stage local MVP toward a normal-user experience:

```text
open Resonova -> connect Spotify Premium -> generate/listen directly
```

The current product is strong enough as an owner/developer MVP. It is not yet v0.1-ready for normal users because access/onboarding, memory-promise honesty, one replay data bug, and blind Spotify playback verification still need decisions or fixes.

This roadmap does not approve billing, public launch, cloud accounts, hosted memory architecture, multi-source expansion, PWA/native packaging, or profile/feedback coding before design.

The roadmap must still preserve the origin goal: Resonova should become a personal AI radio companion that understands the user better over time. Direct browser access is the v0.1 entry path, not a decision to make the product stateless.

## Success Definition

v0.1 succeeds when:

- A normal target user can use Resonova without cloning the repo, running a local server, or bringing AI API keys.
- The product promise is honest: it either has minimal memory/feedback or avoids claiming memory it does not have.
- The core journey works: connect -> generate -> listen -> replay.
- Saved casts replay their own generated assets.
- The product direction remains clear: repeated use, Spotify-authorized trails, saved casts, and feedback should compound into a better personal profile.
- Spotify playback failures are observable enough that boss/chef can distinguish real blockers from recoverable edge cases.
- Parked future work stays parked unless boss explicitly reopens it.

## Phase 1 - Decide Access Shape

Owner: Boss + Chef

Status: accepted as planning direction.

Goal:

- Decide whether v0.1 targets a private hosted owner/beta instance where users connect Spotify and do not bring AI API keys.

Default recommendation:

- Use private hosted owner/beta as the v0.1 north star.
- Keep public accounts, billing, subscriptions, and hosted memory architecture parked.

Decision questions:

- Is v0.1 private/invite-only rather than public launch?
- Who pays AI/TTS costs during validation?
- Is Spotify the only required source for v0.1?
- What is the minimum tester audience: boss only, small trusted testers, or broader private beta?
- Should landing/README copy be softened now, or should minimal memory be prioritized first?

Exit gate:

- Boss and chef agree on the v0.1 access target. Current accepted planning direction: private hosted owner/beta instance.
- No manager starts release architecture until this gate is explicit.

## Phase 3 - Playback Evidence Gate

Owner: Boss real-phone test + chef/technical QA interpretation.

Goal:

- Verify blind Spotify playback with real evidence before assigning more recovery code.

Required evidence:

- One real-phone Diag/Copy-Timeline capture when playback stalls or behaves silently.
- Identify whether the failure is device loss, accepted-but-silent playback, offline/network, or page lifecycle.

Default rule:

- Do not add more recovery branches until this evidence exists.

Exit gate:

- Chef can write a bounded technical manager brief if the failure class is clear.
- If no failure reproduces after reasonable testing, park recovery work and continue product roadmap.

## Phase 4 - Persistent Profile, Spotify Trails, And Feedback Design

Owner: OCP Workspace Lead or equivalent planner under chef gate.

Goal:

- Design the first inspectable memory layer before implementation.
- Keep the original product baseline alive: Resonova should use repeated sessions, explicit feedback, saved casts, and authorized Spotify trails to know the user better over time.
- Define the smallest version that can make a new browser user feel Resonova starts learning after Spotify authorization, without requiring self-hosting or user-managed AI keys.

Scope:

- Minimal taste profile data model.
- Spotify trail inputs that are already available or reasonable to request, including recent plays, top tracks/artists, user playlists, playlist context, and saved-cast history.
- Candidate additional personal-data APIs, including saved library and followed artists, evaluated by value, consent burden, and Spotify policy fit.
- Feedback attachment level: episode, segment, or both.
- Profile update policy: manual, automatic, or semi-automatic.
- Prompt integration approach.
- Owner inspection/edit/reset behavior.
- Open boss decisions.

No-go:

- No code implementation in the design task.
- No cloud or multi-user architecture.
- No billing/subscription work.

Exit gate:

- Chef accepts a design handoff that makes implementation decision-complete.
- Boss approves any product-facing memory behavior before build.

Manager brief:

- `docs/handoffs/Persistent Profile Spotify Trails Design Brief.md`

## Phase 5 - v0.1 Build Decision

Owner: Boss + Chef

Start this only after Phases 1-4 produce enough evidence.

Possible implementation lanes:

- Private hosted beta access.
- Minimal memory/feedback implementation.
- Minimal Spotify trail ingestion into an inspectable profile.
- Playback stall verification fix, if Phase 3 proves it is needed.
- Final release-readiness QA pass.

Exit gate:

- Boss accepts the v0.1 scope.
- Chef prepares manager briefs with exact file boundaries, validation commands, and handoff requirements.

## Parked Until Reopened

- Public launch.
- Billing, subscriptions, quotas, payments, or free trials.
- Public account system.
- Full multi-user hosted memory storage architecture.
- Multi-source support beyond Spotify.
- Native app or PWA packaging.
- Interactive "talk to your hosts" mode.
- Broad mobile recovery rewrites or retry loops.
- Self-hosting as the main product promise.
